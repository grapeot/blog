# Deprioritized: 每篇文章的抽象题图

这份文档记录一个暂缓推进的方向：给 Computing Life 的每篇文章增加 light/dark 抽象题图。当前判断是：这个方向有潜在价值，但不进入近期实现队列。主站刚完成 Signal redesign，首页的信息密度和工程笔记气质比视觉丰富度更重要。

## 当前结论

题图的合理角色是文章的情绪索引，用来帮助读者扫首页、文章页和社交预览时快速区分主题。它不应该承担解释文章内容的任务，也不应该把站点推向封面流或内容营销站。

近期不接入模板，也不批量生成旧文图片。后续如果重新启动，应该先做少量文章的视觉系统试验，再决定是否进入文章页或首页列表。

## 为什么暂缓

1. Signal redesign 的核心收益来自信息结构、阅读体验和双语入口，而不是图片系统。继续加题图会把注意力从已验证的文字索引体验转移到新的视觉工作流。
2. 每篇文章维护 light/dark 两张图会引入内容模型、压缩、alt、裁切、预览和回归检查。这个成本只有在题图明显改善扫描效率时才成立。
3. AI 图像容易把个人工程博客带向泛化的科技海报风格。即使单张图好看，批量出现在首页也可能削弱长期工程笔记的可信感。
4. 当前还没有证据说明小题图能比标题、summary、category 和 tag 更有效地帮助读者选择文章。

## 如果未来重启，题图要解决什么问题

文章列表现在主要靠标题、summary、category、tag 传达信息。这对文字读者有效，但连续扫十篇时，所有文章的视觉重量接近，读者需要逐行读标题才能建立主题差异。

题图的目标是提供第二条通道：让主题先被感知，再被文字确认。健康数据、前端系统、AI 工作流、天文模拟和生活反思可以有不同的视觉 silhouette。这样做只有在降低识别成本时才有意义。

这里的关键约束是显示尺寸很小，常见尺寸大约是 128×128，最多 256×256。所以题图必须按小图标设计，而不是按海报设计。复杂纹理、细碎线条和多层叙事在大图里可能好看，缩到 128px 后会变成噪声。成功标准是缩小以后仍然保留清楚的情绪和主题 silhouette。

## 不应该做成什么

第一，题图不是文章内配图。文章内配图解释具体内容，题图只负责入口识别。Gaia 文章里的真实渲染图可以出现在正文，但首页题图可以更抽象，不必复刻最终银河图。

第二，题图不应该是 AI 生成海报。大标题、人物、赛博背景、发光球体、拟物插画，会让博客显得像内容营销站。Computing Life 的气质是工程笔记，题图要克制。

第三，题图不应该一篇一个完全不同风格。长期看，统一视觉语法比单篇惊艳更重要。读者应该感到这些图来自同一个系统。

第四，题图不应该依赖渐变和大面积光效。渐变很容易把图做成廉价科技海报，缩小到 128px 后还会变成脏色块。题图应该用纯色、清楚边界和少量颜色关系表达情绪。

## 视觉语言：抽象但有语义

每张题图可以拆成两层：主题形状和情绪色块。

主题形状负责领域识别。健康文章可以用曲线、节律、点阵和时间轴；前端系统可以用盒子、流式布局、基线和测量尺；AI 工作流可以用节点、路径、文件夹、箭头和记忆层；天文可以用星点、密度场、坐标网格和圆盘结构；生活反思可以用日历、轨迹、家居轮廓或低频波形。

情绪色块负责文章感觉。失眠文章可以是冷色夜晚里的一条暖色恢复曲线；前端 trade-off 可以是秩序和断裂并存；Skill 文章可以是散乱步骤被整理成路径；Gaia 文章可以是星点密度逐渐形成结构。

一致性来自固定 palette、纯色块、低细节密度和几何构图，而不是纹理或光效。可以有少量细线和点，但不要依赖 grain、glow、blur、neon 或渐变。

## Palette

如果未来继续生成，先固定 palette，再写 prompt。否则 GPT-Image-2 很容易让每篇图漂到不同美术风格。

| 用途 | Light | Dark |
|---|---|---|
| 背景 | `#F8F9FB` paper | `#0D0F14` blue-black |
| 主形状 | `#2547E0` cobalt | `#8EA2FF` mist cobalt |
| 次级形状 | `#A9B2C3` slate | `#394256` deep slate |
| 暖色点缀 | `#D99A24` muted amber | `#D99A24` muted amber |
| 软高亮 | `#FFF2C2` cream | `#FFF2C2` cream，小面积使用 |

生成时要明确禁止 gradient、glow、neon、lens flare、photorealistic lighting、metallic reflection。允许 anti-aliasing，但视觉上必须是 flat vector-like color blocks。

## Light/Dark 处理

如果题图进入实现，建议每篇生成 light/dark 两张图。浅色版用低饱和背景、深蓝线条、轻暖色点缀；深色版用深蓝黑背景、浅蓝线条、同一暖色点缀。

关键流程是两阶段生成：先生成 light 版，再把 light 图作为输入，让 GPT-Image-2 只改成 dark palette。不要让 light 和 dark 用两个完全独立的 text-to-image prompt 抽样，否则主题切换时会出现构图跳变。dark prompt 必须明确要求保留 composition、silhouette、shape count、geometry 和 visual metaphor，只改颜色。

## 文章方向示例

| 文章 | 主题判断 | 抽象图方向 |
|---|---|---|
| `AI如何导致和修复了我的失眠问题` | 健康数据 + AI 分析 + 生活实验 | 夜色背景里的睡眠阶段带、心率/咖啡因/活动点阵，多变量线条最终汇入一条暖色恢复曲线 |
| `一行代码的事，Web 为什么做了三十年还没做到` | 前端布局 + 系统抽象 + trade-off | 文本块在容器中重排，旁边有灰色中间层；另一侧是一条清晰测量尺，表达可查询与全局重排的冲突 |
| `用好AI的第二步：先写Skill再执行` | AI 工作流 + 外化 + 复用 | 散乱任务点被写入一个 skill 卡片，再变成可复用路径；视觉中心是规则被外化成结构 |
| `把18亿颗星星画在一张图上，能还原我们拍到的银河吗？` | 天文数据 + 模拟翻车 + 物理结构 | 从随机星点到有密度梯度的银河盘，带坐标网格；避免漂亮星空壁纸 |
| `为什么AI只会说正确的废话，以及怎么把它逼出舒适区` | AI 上下文 + 认知结构 + 非共识输出 | 平均化的灰蓝 consensus 云，被个人 context 节点拉出一条偏离主流的路径 |

这些例子说明一个原则：题图不画 AI、Web、健康这些名词本身，而画文章里的核心关系。失眠文章不是画床，前端文章不是画浏览器，Skill 文章不是画机器人，Gaia 文章不是画唯美星空。

## 生成方法

未来如果重启，先不要一次性给所有旧文生成。应该选 8-12 篇近期代表文章做视觉系统试验，覆盖几类主题：AI 方法论、系统设计、天文、健康、生活反思、硬件/咖啡。

每篇生成前先写一个视觉 brief，包含四项：主题关系、禁止元素、形状语言、light/dark 色彩。brief 比直接 prompt 更重要，因为它能把视觉系统维持住。

示例 brief：

```text
文章：AI如何导致和修复了我的失眠问题
主题关系：夜间 AI 高强度思考 -> 健康数据采集 -> 多变量分析 -> 睡眠恢复
禁止元素：不要人物、床、医生、脑袋发光、赛博朋克机器人
形状语言：时间轴、睡眠阶段带、散点、回归线、一个恢复曲线
色彩：light 版为冷灰纸底 + 钴蓝线条 + 暖琥珀恢复线；dark 版为深蓝黑底 + 雾蓝线条 + 暖琥珀恢复线
```

生成模型先用 GPT-Image-2 出位图，人工/AI 筛选后再做裁切和压缩。SVG 暂时不作为主路线，只保留为未来可能的设计系统探索。

## 内容模型的可能形态

这里要遵守 `docs/design.md` 的原则：展示需求不要过度污染内容源文件。

最小方案是在文章 frontmatter 里加可选字段：

```yaml
HeroLight: images/heroes/ai-sleep-light.webp
HeroDark: images/heroes/ai-sleep-dark.webp
HeroAlt: Abstract sleep data timeline converging into a recovery curve
```

模板逻辑是：如果有 hero，就在文章页标题附近和首页列表中使用；如果没有，就保持纯文字布局。这样可以渐进加入，不必一次性覆盖所有文章。

另一种更少污染 frontmatter 的方案是约定路径：根据 slug 自动找 `content/images/heroes/{slug}-light.webp` 和 `{slug}-dark.webp`。如果文件存在就显示。这个方案更干净，但 Pelican/Jinja 里检查文件存在会更麻烦，也会让内容和资源之间的关系变隐式。

当前不选择任何实现方案。等视觉系统有明确收益后，再决定 frontmatter、manifest 或约定路径。

## 首页显示原则

题图不应该把首页重新变成卡片海报流。当前首页的价值是高密度索引，题图要服务这个索引，而不是替代它。

可以考虑三种密度：

1. 小 thumbnail。放在每篇文章左侧或标题上方，尺寸克制，作为主题标记。
2. 重点文章大图，其余文章小图。只给最新一篇或每页第一篇更大视觉权重。
3. 只在文章页和 social preview 用题图，首页仍保持文字列表。

如果未来重启，优先试第一种：小而一致的 abstract thumbnail。它符合变化信息优先的原则，也不会让固定视觉系统压过文章标题。

## 重启条件

这条路线只有在满足以下条件时再进入实现：

1. 有 8-12 篇文章的视觉系统试验，并且 128px contact sheet 下仍能稳定识别主题。
2. light/dark 两张图通过 input edit 保持构图一致。
3. 首页信息密度没有明显下降。
4. 图片文件大小、lazy loading、alt、移动端裁切都有清楚方案。
5. 视觉效果仍然像工程笔记的索引，而不是 AI 生成封面流。

判断成功的标准不是图好看，而是看一屏 10 篇文章时，读者能更快区分主题，并且不会觉得站点变成封面流。
